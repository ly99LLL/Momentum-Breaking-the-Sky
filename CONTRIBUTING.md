## 如何贡献

### 报告 Bug

1. 在 [Issues](../../issues) 中搜索是否已存在相同报告
2. 使用 **Bug 报告** 模板创建 Issue
3. 附上：操作系统版本、GPU 型号、复现步骤、期望行为 vs 实际行为

### 提出新功能

1. 在 [Issues](../../issues) 中搜索是否已有类似提案
2. 使用 **功能请求** 模板创建 Issue
3. 说明：这个功能解决什么问题、建议的实现方式

### 提交代码

1. Fork 本仓库
2. 创建功能分支：`git checkout -b feature/your-feature`
3. 遵循[代码规范](#代码规范)编写代码
4. 确保测试通过
5. 提交 Pull Request，使用 PR 模板

## 开发流程

本项目严格遵循 **PRD 驱动工作流**：

1. **Phase 0 · 定位** — 确认需求与技术决策
2. **Phase 1 · 设计** — 定义接口、编写测试用例
3. **Phase 2 · 实现** — 一次一个模块，测试通过再继续
4. **Phase 3 · 验证** — 独立运行每个模块
5. **Phase 4 · 记录** — 更新文档、提交 commit

详见 [`CLAUDE.md`](CLAUDE.md)。

## 代码规范

### 通用纪律

- **单文件 ≤ 300 行**，超过必须拆分
- **每个函数必须注释**：做什么、输入什么、返回什么
- **禁止全局可变状态**，状态封装在参数/返回值或类实例中
- **所有外部依赖记录在 `docs/architecture.md` §16**
- **错误必须处理**，不允许静默吞异常

### 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件名 | 小写 + 连字符 | `gesture-detector.js` |
| 函数/变量 | 驼峰命名 | `readSensorData` |
| 常量 | 全大写 + 下划线 | `MAX_RETRY_COUNT` |
| 类/组件 | 大驼峰命名 | `GestureDetector` |

### 目录纪律

- 代码文件在 `src/` 对应子目录下，禁止堆在根目录
- 每个模块自包含：代码 + 测试镜像放置
- 模块间通过 EventBus 通信，不直接 import 对方实现

## 提交规范

### Commit 消息格式

```
[模块名] 做了什么
```

示例：
```
[input] MediaPipe 姿态检测接入通过测试
[render] WebGPU raymarch 着色器完成
[state] 势能累积算法校准完成
```

禁止一个大 commit 包含多个模块的改动。

### 分支命名

| 分支类型 | 命名格式 | 示例 |
|---------|---------|------|
| 功能 | `feature/描述` | `feature/tear-mechanics` |
| 修复 | `fix/描述` | `fix/camera-permission` |
| 文档 | `docs/描述` | `docs/api-reference` |
